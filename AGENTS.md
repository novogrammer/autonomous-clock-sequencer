# AGENTS.md

## このリポジトリについて

このリポジトリは、Webブラウザで動作するシーケンサーの実験・実装を扱う。  
最終的には、複数端末が同じURLを開くことで、通信なしに近い条件で同じ時間軸・同じ記述を共有できる構成を目指す。

ただし、現時点では最終形を一気に作らず、段階的に進める。  
まずは Phase 0 として、メトロノームのみで時間モデルを検証する。

---

## 技術方針

- UI: React
- 状態管理: Zustand
- 音声再生: Tone.js
- TypeScript を使う
- 再生中の通信は行わない
- 状態共有は URL を使う
- QRコードは後で URL 配布手段として扱う
- URL は当面圧縮しない

---

## 設計方針

### 1. 時間モデルを中核にする
このアプリの中心は、絶対時刻基準で再生位置を計算すること。  
Unix epoch から続く共通グリッドとして、現在位置を都度計算する。

### 2. UI と音声エンジンを分離する
React / Zustand 側に Tone.js の生インスタンスや AudioNode を深く持たない。  
シリアライズしやすい状態と、実行オブジェクトを分ける。

### 3. URL と相互変換しやすい state を保つ
共有対象になる state は URL に載せやすい形を優先する。  
一時的な内部状態や実行中オブジェクトは URL に載せない。

### 4. 後からシーケンサーへ拡張できる構造にする
Phase 0 はメトロノームだけだが、後で `pattern` や `kit` を追加できる構造にする。  
メトロノーム専用の作りに寄せすぎない。

---

## 期待する責務分離

### Transport 層
- 絶対時刻から拍位置や step 位置を計算する
- `bpm`, `stepsPerBeat`, `swing` を扱う
- 音は鳴らさない

### Scheduler / Engine 層
- Tone.js を使って音を予約・再生する
- 先読みスケジューリングを担当する
- React state に依存しすぎない

### State 層
- Zustand を使う
- URL と相互変換しやすい値を保持する
- シリアライズ不能な実行オブジェクトは持たない

### UI 層
- 検証しやすさを優先する
- 初期段階では見た目の作り込みより構造の明瞭さを重視する

## 将来の pattern / kit 方針

現時点では Phase 0 の実装対象ではないが、将来前提として以下を維持する。

### kit
- `kit` は URL の解釈ルール
- トラック数、トラック順、各トラックの意味を定義する
- 互換性が壊れる変更を行う場合、別 ID にする

### pattern
- `pattern` は 1 つの文字列
- 各トラックは kit で定義された順番に並ぶ
- 各トラックは `0/1` 文字列
- 区切りは `_`

例:
`pattern=1000100010001000_0000100000001000_1010101010101010`

### pattern 読み込み時
- 足りない場合は補完する
- 長すぎる場合は切り捨てる
- 不正文字は警告しつつ `0` 扱いにする
- エラーで停止しない

---

## 時間モデル方針

### 時間基準
- 再生位置は Unix epoch から続く共通グリッドとして計算する
- 再生開始時刻を state や URL には持たない
- 各端末は現在の Unix time ミリ秒から現在位置を計算する

### BPM 変更
- BPM変更は即時反映
- Unix time 基準の共有グリッドを優先する
- 既存の先読み予約は捨てて再スケジュールする

### swing
- `swing=0` はストレート
- `swing>0` は swing 有効
- 値は swing の強さを表す

## 実装時の注意

- `history.pushState()` を乱用しない
- 通常の URL 更新は `history.replaceState()` を使う
- 毎操作で履歴を増やさない
- Tone.js の unlock を考慮する
- スマホでの最低限の利用を意識する
- 過度な最適化や過剰設計は不要
- まず動く最小単位を作る
