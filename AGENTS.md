# AGENTS.md

## このリポジトリについて

このリポジトリは、Webブラウザで動作するシーケンサーの実験・実装を扱う。  
最終的には、複数端末が同じ URL を開くことで、通信なしに近い条件で同じ時間軸・同じ記述を共有できる構成を目指す。

現行仕様や Phase ごとの詳細は `docs/` を参照する。

- 全体概要: `docs/overview.md`
- 時間モデル: `docs/spec/time-model.md`
- URL / state 仕様: `docs/spec/url-state.md`
- 現行 Phase: `docs/phases/phase-0.md`
- 次 Phase 案: `docs/phases/phase-1.md`

---

## 技術方針

- UI: React
- 状態管理: Zustand
- 音声再生: Tone.js
- TypeScript を使う
- 再生中の通信は行わない
- 状態共有は URL を使う
- QR コードは後で URL 配布手段として扱う
- URL は当面圧縮しない

---

## 設計方針

### 1. 時間モデルを中核にする
このアプリの中心は、絶対時刻基準で再生位置を計算すること。  
Unix epoch から続く共通グリッドとして、現在位置を都度計算する。

### 2. UI と音声エンジンを分離する
React / Zustand 側に Tone.js の生インスタンスや AudioNode を深く持たない。  
シリアライズしやすい状態と、実行オブジェクトを分ける。

### 3. URL と相互変換しやすい state を保つ
共有対象になる state は URL に載せやすい形を優先する。  
一時的な内部状態や実行中オブジェクトは URL に載せない。

### 4. 後からシーケンサーへ拡張できる構造にする
現行 Phase は最小構成だが、後で `pattern` や `kit` を追加できる構造にする。  
一時的な画面専用実装に寄せすぎない。

---

## 期待する責務分離

### Transport 層
- 絶対時刻から拍位置や step 位置を計算する
- `bpm`, `stepsPerBeat`, `swing` を扱う
- 音は鳴らさない

### Scheduler / Engine 層
- Tone.js を使って音を予約・再生する
- 先読みスケジューリングを担当する
- React state に依存しすぎない

### State 層
- Zustand を使う
- URL と相互変換しやすい値を保持する
- シリアライズ不能な実行オブジェクトは持たない

### UI 層
- 検証しやすさを優先する
- 初期段階では見た目の作り込みより構造の明瞭さを重視する
- 表に見える表示には Phase 名を出さない

---

## 実装時の注意

- `history.pushState()` を乱用しない
- 通常の URL 更新は `history.replaceState()` を使う
- 毎操作で履歴を増やさない
- Tone.js の unlock を考慮する
- スマホでの最低限の利用を意識する
- 過度な最適化や過剰設計は不要
- まず動く最小単位を作る

---

## 作業の進め方

- まず現在の Phase の範囲を守る
- 将来の構想を見越して構造を整える
- ただし将来機能を先回りして実装しすぎない
- 不明点があれば、既存仕様を壊さない側に寄せる

---

## 命名・コード方針

- 名前は説明的でよい
- 略語は必要最小限にする
- URL パラメーター名は既存方針を尊重する
- state 名は URL 名とできるだけ対応させる
- シリアライズ可能な値と実行時オブジェクトを混ぜない

---

## 目標

このリポジトリでは、最終的な作品やアプリの前に、  
時間基準・URL共有・自律再生という中核構造を壊さずに育てていくことを重視する。
